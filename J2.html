📦 Installation & configuration de React Navigation
💡 Analogie

React Navigation, c'est le GPS de ton application mobile. Comme un GPS gère les routes entre des destinations, React Navigation gère les transitions entre les écrans. Il sait quel écran est affiché, où tu viens, et comment revenir en arrière — exactement comme le bouton retour de ton téléphone.

⚠️Toujours utiliser npx expo install au lieu de npm install pour les bibliothèques Expo ! Cette commande installe la version compatible avec ta version d'Expo. npm install pourrait installer une version trop récente ou trop ancienne qui créerait des bugs difficiles à déboguer.
Installer le cœur de React Navigation (indispensable, donne le contexte de navigation) :
npx expo install @react-navigation/native
Installer les dépendances natives requises par React Navigation (gestion des zones sécurisées, performances) :
npx expo install react-native-screens react-native-safe-area-context
Installer le Stack Navigator (navigation en pile, avec header et bouton retour) :
npm install @react-navigation/native-stack
Installer le Tab Navigator (barre d'onglets en bas de l'écran) :
npm install @react-navigation/bottom-tabs
Les icônes Ionicons sont déjà incluses dans Expo — aucune installation supplémentaire :
import {'{'} Ionicons {'}'} from '@expo/vector-icons';
App.js — point d'entrée avec NavigationContainer
// ─── IMPORTS ───────────────────────────────────────────────────────────────
// NavigationContainer : le "conteneur" obligatoire qui active la navigation
// dans toute l'app. Il doit englober TOUT le reste. Une seule fois dans App.js
import { NavigationContainer } from '@react-navigation/native';

// AppNavigator : notre propre fichier qui définit tous les écrans et navigators
// On crée ce fichier dans navigation/AppNavigator.js
import { AppNavigator } from './navigation/AppNavigator';

// ─── COMPOSANT PRINCIPAL ────────────────────────────────────────────────────
// C'est ici que React Native démarre l'app (App.js = fichier d'entrée)
export default function App() {
  return (
    // NavigationContainer DOIT entourer tout — c'est lui qui fournit
    // le contexte de navigation à tous les composants enfants.
    // Sans lui → erreur : "Couldn't find a navigation object"
    <NavigationContainer>
      // AppNavigator contient notre stack/tab — on l'importe depuis
      // navigation/AppNavigator.js pour garder App.js propre
      <AppNavigator />
    </NavigationContainer>
  );
}
🚫Erreur classique débutant : mettre NavigationContainer à l'intérieur d'un écran ou d'un composant enfant. Il doit être dans App.js et n'apparaître qu'une seule fois dans toute l'application.
📚 Stack Navigator — navigation en pile d'écrans
💡 Analogie

Le Stack Navigator fonctionne comme une pile d'assiettes. Tu poses une assiette (navigate → nouvel écran). Tu enlèves l'assiette du dessus (goBack → écran précédent). Tu ne peux accéder qu'à l'assiette du haut — les autres sont "en dessous" dans l'historique. Sur iOS c'est le swipe depuis le bord gauche, sur Android c'est le bouton retour physique.

navigation/AppNavigator.js — créer un Stack Navigator
// ─── IMPORTS ────────────────────────────────────────────────────────────────
// createNativeStackNavigator : fonction qui crée un "objet Stack"
// Cet objet contient deux composants qu'on utilisera : Stack.Navigator et Stack.Screen
import { createNativeStackNavigator } from '@react-navigation/native-stack';

// On importe les composants-écrans depuis nos fichiers locaux
import { HomeScreen }   from '../screens/HomeScreen';
import { DetailScreen } from '../screens/DetailScreen';

// ─── CRÉATION DU STACK ──────────────────────────────────────────────────────
// createNativeStackNavigator() retourne un objet avec deux composants :
//   Stack.Navigator → le conteneur qui gère la pile d'écrans
//   Stack.Screen    → chaque écran individuel dans la pile
const Stack = createNativeStackNavigator();

// ─── COMPOSANT DU NAVIGATOR ──────────────────────────────────────────────────
// On exporte ce composant pour l'utiliser dans App.js
export function AppNavigator() {
  return (
    // Stack.Navigator : gère la pile — initialRouteName = premier écran affiché
    // screenOptions : options communes à TOUS les écrans (style du header)
    <Stack.Navigator
      initialRouteName="Home"
      screenOptions={{
        // Couleur de fond du header (barre de navigation en haut)
        headerStyle: { backgroundColor: '#0f172a' },
        // Couleur du texte et de l'icône de retour dans le header
        headerTintColor: '#f1f5f9',
        // Style du titre affiché dans le header
        headerTitleStyle: { fontWeight: '700' },
      }}
    >
      // Stack.Screen : déclare un écran dans la pile
      // name="Home"       → le nom qu'on utilisera dans navigate('Home')
      // component={...}   → le composant React à afficher pour cet écran
      // options={{...}}   → options spécifiques à CET écran (ex: titre du header)
      <Stack.Screen
        name="Home"
        component={HomeScreen}
        options={{ title: 'Accueil' }}
      />
      <Stack.Screen
        name="Detail"
        component={DetailScreen}
        options={{ title: 'Détail' }}
      />
    </Stack.Navigator>
  );
}

// ════════════════════════════════════════════════════════════
// ÉCRAN A — HomeScreen.js
// Le composant reçoit automatiquement la prop "navigation"
// grâce au fait qu'il est déclaré dans un Stack.Screen
// ════════════════════════════════════════════════════════════
const HomeScreen = ({ navigation }) => (
  <TouchableOpacity
    onPress={() =>
      // navigate('Detail') → React Navigation empile DetailScreen
      // Le 2ème argument est un objet : les "paramètres" envoyés à l'écran
      // Accessible dans DetailScreen via route.params
      navigation.navigate('Detail', {
        themeId: 'T1',
        title:   'Composants & JSX',
        count:   6,
      })
    }
  >
    <Text>Voir le thème T1</Text>
  </TouchableOpacity>
);

// ════════════════════════════════════════════════════════════
// ÉCRAN B — DetailScreen.js
// Ce composant reçoit aussi navigation (pour goBack)
// ET route (qui contient les paramètres envoyés par A)
// ════════════════════════════════════════════════════════════
const DetailScreen = ({ route, navigation }) => {
  // Déstructuration des paramètres : on extrait les valeurs de route.params
  // route.params contient l'objet passé dans navigate('Detail', { ... })
  const { themeId, title, count } = route.params;

  return (
    <View>
      // On utilise les paramètres reçus pour afficher le contenu
      <Text>{'{title}'} ({'{themeId}'}) — {'{count}'} exercices</Text>
      <TouchableOpacity
        // goBack() : dépile l'écran courant → retour à HomeScreen
        onPress={() => navigation.goBack()}
      >
        <Text>← Retour</Text>
      </TouchableOpacity>
    </View>
  );
};
// ─── Option 1 : titre statique (défini une fois pour toujours)
<Stack.Screen
  name="Detail"
  component={DetailScreen}
  options={{ title: 'Détail', headerBackTitle: 'Retour' }}
// headerBackTitle : texte du bouton retour (iOS uniquement)
/>

// ─── Option 2 : titre dynamique basé sur les paramètres de navigation
// options reçoit une fonction qui a accès à "route" (avec les params)
<Stack.Screen
  name="Detail"
  component={DetailScreen}
  options={({ route }) => ({
    // Le titre du header est le titre passé en paramètre
    // Ex : navigate('Detail', { title: 'Composants' }) → header affiche 'Composants'
    title: route.params.title,
  })}
/>

// ─── Option 3 : modifier le header depuis le composant lui-même
// Utile quand le titre dépend de données chargées après le montage
useEffect(() => {
  // setOptions() met à jour les options du header en temps réel
  // Ici on change le titre quand les données "data" ont été chargées
  navigation.setOptions({ title: data.name });
}, [data]); // ← dépendance : se re-exécute quand data change

🗂 Bottom Tab Navigator — barre d'onglets en bas
💡 Analogie

Le Tab Navigator, c'est exactement la barre de navigation en bas d'Instagram, YouTube ou WhatsApp. Chaque onglet est un "monde" indépendant. La différence avec le Stack : passer d'un onglet à l'autre est instantané car les écrans restent en mémoire une fois visités — pas besoin de les reconstruire.

navigation/TabNavigator.js — Tab avec icônes Ionicons
// ─── IMPORTS ────────────────────────────────────────────────────────────────
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';
// Ionicons : bibliothèque d'icônes vectorielles incluse avec Expo
import { Ionicons } from '@expo/vector-icons';

// createBottomTabNavigator() → comme createNativeStackNavigator()
// retourne un objet avec Tab.Navigator et Tab.Screen
const Tab = createBottomTabNavigator();

export function TabNavigator() {
  return (
    <Tab.Navigator
      // screenOptions : fonction appelée pour chaque onglet
      // "route" = infos sur l'onglet courant (route.name = son nom)
      screenOptions={({ route }) => ({

        // tabBarIcon : définit l'icône à afficher dans la barre
        // focused : true si l'onglet est actif, false sinon
        // color : la couleur à utiliser (active ou inactive, définie plus bas)
        // size : la taille de l'icône recommandée par React Navigation
        tabBarIcon: ({ focused, color, size }) => {
          // On choisit l'icône selon l'onglet ET son état actif/inactif
          // Convention Ionicons : le nom sans "-outline" = icône pleine (actif)
          //                       le nom avec "-outline" = icône vide (inactif)
          const icons = {
            Accueil:  focused ? 'home'           : 'home-outline',
            Explorer: focused ? 'compass'        : 'compass-outline',
            Profil:   focused ? 'person'         : 'person-outline',
          };
          // On retourne l'icône Ionicons avec le bon nom, la bonne taille et couleur
          return <Ionicons name={icons[route.name]} size={size} color={color} />;
        },

        // tabBarActiveTintColor : couleur de l'onglet ET de l'icône quand actif
        tabBarActiveTintColor: '#34d399',
        // tabBarInactiveTintColor : couleur quand l'onglet n'est pas sélectionné
        tabBarInactiveTintColor: '#64748b',
        // tabBarStyle : style de la barre d'onglets elle-même (fond, bord)
        tabBarStyle: {
          backgroundColor: '#1e293b',
          borderTopColor:   '#334155',
        },
        // Même style de header que dans le Stack pour la cohérence visuelle
        headerStyle:      { backgroundColor: '#0f172a' },
        headerTintColor:  '#f1f5f9',
      })}
    >
      // Chaque Tab.Screen = un onglet dans la barre
      // name : le nom affiché sous l'icône ET utilisé dans navigate()
      <Tab.Screen name="Accueil"  component={HomeScreen}    />
      <Tab.Screen name="Explorer" component={ExploreScreen} />
      <Tab.Screen name="Profil"   component={ProfileScreen}  />
    </Tab.Navigator>
  );
}
🔀 Navigation imbriquée — Stack à l'intérieur d'un Tab
💡C'est le pattern le plus courant dans les apps réelles : chaque onglet a sa propre Stack. Naviguer de ListeThemes → DetailTheme reste dans l'onglet — l'utilisateur ne change pas d'onglet, il navigue "à l'intérieur".
navigation/index.js — Stack imbriqué dans un onglet du Tab
// ─── IMPORTS ────────────────────────────────────────────────────────────────
const Tab       = createBottomTabNavigator();
// On crée un Stack SÉPARÉ pour l'onglet Accueil
// (si on utilisait le même Stack que le reste, les écrans se mélangeraient)
const HomeStack = createNativeStackNavigator();

// ─── STACK DE L'ONGLET ACCUEIL ──────────────────────────────────────────────
// Ce composant représente TOUT ce qu'on peut faire dans l'onglet Accueil
// Il a sa propre pile d'écrans, indépendante du Tab
function HomeStackNavigator() {
  return (
    <HomeStack.Navigator>
      // L'écran de départ dans cet onglet : la liste des thèmes
      <HomeStack.Screen name="HomeList"  component={HomeScreen}   />
      // Écran de détail accessible depuis HomeList
      // navigate('Detail', {...}) depuis HomeScreen atterrit ici
      <HomeStack.Screen name="Detail"    component={DetailScreen} />
    </HomeStack.Navigator>
  );
}

// ─── NAVIGATEUR PRINCIPAL ───────────────────────────────────────────────────
// Le Tab Navigator qui contient nos onglets
// L'onglet Accueil utilise HomeStackNavigator comme composant
// → tout l'onglet Accueil EST la Stack définie au-dessus
export function RootNavigator() {
  return (
    <Tab.Navigator>
      // component={HomeStackNavigator} : l'onglet Accueil = la Stack complète
      // Naviguer de HomeList → Detail reste dans cet onglet, sans changer d'onglet !
      <Tab.Screen name="Accueil" component={HomeStackNavigator}  />
      <Tab.Screen name="Profil"  component={ProfileScreen}       />
    </Tab.Navigator>
  );
}

S4 · useEffect & Fetch API

useEffect · fetch · async/await · 3 états loading/error/data · FlatList dynamique

⚡ useEffect — Déclencher du code après le rendu
💡 Analogie

useEffect, c'est comme un employé qui attend que la boutique soit ouverte pour faire son travail. React ouvre d'abord le "magasin" (affiche le composant), puis l'employé commence à travailler (exécute l'effet). Sans ça, si tu lançais un fetch pendant le rendu, React s'affolerait à chaque re-render — boucle infinie garantie.

1

Le composant se rend une première fois — souvent vide ou avec un spinner

React affiche d'abord la structure avec les données initiales : data=null, loading=true. L'utilisateur voit un indicateur de chargement. C'est voulu — on affiche quelque chose immédiatement.

2

useEffect s'exécute après ce premier rendu

Le code dans useEffect tourne après que l'écran est affiché. Le tableau [] en 2ème argument signifie "exécuter une seule fois, au montage". Sans ce tableau : exécuté à chaque re-render → boucle infinie.

3

Les données arrivent → setState() → React re-render automatiquement

Quand le fetch réussit, on appelle setData(json). React détecte le changement d'état et re-render le composant avec les nouvelles données. L'utilisateur voit la liste s'afficher.

4

Le tableau de dépendances contrôle quand ré-exécuter l'effet

[] = une seule fois au montage (fetch initial). [id] = re-exécuter chaque fois que id change (ex: charger les détails d'un autre item). Pas de tableau = à chaque re-render (⚠ dangereux !).

Pattern complet — useEffect + fetch + 3 états
// ─── IMPORTS ────────────────────────────────────────────────────────────────
// On importe useState ET useEffect depuis React
import { useState, useEffect } from 'react';
import { View, Text, FlatList, ActivityIndicator } from 'react-native';

const ListeItems = ({ id }) => {

  // ─── 3 ÉTATS OBLIGATOIRES ────────────────────────────────────────────────
  // Ces 3 états couvrent tous les cas possibles d'un fetch :
  //   data    : les données récupérées (null au départ, tableau après le fetch)
  //   loading : est-ce qu'on est en train de charger ? (true = afficher spinner)
  //   error   : y a-t-il eu une erreur ? (null = pas d'erreur, sinon le message)
  const [data,    setData]    = useState(null);
  const [loading, setLoading] = useState(true);  // true = commence en chargement
  const [error,   setError]   = useState(null);

  // ─── useEffect ───────────────────────────────────────────────────────────
  // S'exécute APRÈS le premier rendu (et à chaque changement de "id")
  // [id] = tableau de dépendances → "re-exécuter si id change"
  useEffect(() => {
    fetchData(); // on appelle notre fonction async définie ci-dessous
  }, [id]);    // ← dépendance : nouveau id = nouveau fetch

  // ─── FONCTION FETCH ASYNC ────────────────────────────────────────────────
  // On NE peut PAS mettre async directement dans useEffect
  // Donc on définit une fonction async à l'intérieur et on l'appelle
  const fetchData = async () => {
    try {
      setLoading(true);   // début du chargement → afficher le spinner
      setError(null);     // reset l'erreur précédente si on re-essaie

      // fetch() envoie la requête HTTP GET vers l'URL
      // await : on ATTEND que la réponse arrive (sans bloquer l'app)
      const response = await fetch(`https://api.example.com/items/${'{id}'}`);

      // .json() convertit la réponse (texte brut) en objet JavaScript
      // await : on attend aussi cette conversion qui est asynchrone
      const json = await response.json();

      // On met les données dans l'état → React va re-render avec les données
      setData(json);

    } catch (err) {
      // Si fetch() ou .json() lance une erreur, on arrive ici
      // On stocke le message d'erreur pour l'afficher à l'utilisateur
      setError('Impossible de charger les données. Vérifie ta connexion.');

    } finally {
      // finally = TOUJOURS exécuté (que ça réussisse ou non)
      // On arrête le spinner dans tous les cas
      setLoading(false);
    }
  };

  // ─── RENDU CONDITIONNEL ──────────────────────────────────────────────────
  // On affiche quelque chose de différent selon l'état courant

  // CAS 1 : chargement en cours → spinner
  if (loading) return <ActivityIndicator size="large" color="#34d399" />;

  // CAS 2 : erreur → message + bouton pour réessayer
  if (error) return (
    <View>
      <Text style={{ color: '#fca5a5' }}>⚠ {'{error}'}</Text>
      <TouchableOpacity onPress={fetchData}><Text>Réessayer</Text></TouchableOpacity>
    </View>
  );

  // CAS 3 : données chargées → afficher la liste (on arrive ici seulement si
  // loading=false ET error=null)
  return (
    <FlatList
      data={data}
      // keyExtractor : fournit un identifiant unique à chaque élément
      // Obligatoire pour que React puisse optimiser le rendu de la liste
      keyExtractor={(item) => item.id.toString()}
      renderItem={({ item }) => <Text>{'{item.name}'}</Text>}
    />
  );
};

Exemples — cliquer pour développer :

// ─── ÉTATS ───────────────────────────────────────────────────────────────
// items : toutes les données chargées depuis l'API (ne change pas)
// search : texte saisi par l'utilisateur dans le champ de recherche
const [items,  setItems]  = useState([]);
const [search, setSearch] = useState('');

// ─── FILTRAGE ────────────────────────────────────────────────────────────
// On ne modifie pas "items" — on crée une NOUVELLE liste filtrée à chaque rendu
// toLowerCase() sur les deux chaînes pour ignorer la casse (Maj/min)
const filtered = items.filter(item =>
  item.name.toLowerCase().includes(search.toLowerCase())
  // includes() : retourne true si la chaîne contient la valeur recherchée
);

// ─── AFFICHAGE ───────────────────────────────────────────────────────────
<TextInput
  value={search}
  // À chaque frappe : setSearch met à jour l'état → React re-render
  // → filtered est recalculé → la liste se met à jour automatiquement
  onChangeText={setSearch}
  placeholder="Rechercher..."
/>
<FlatList
  // On donne "filtered" à FlatList, pas "items" → seulement les résultats
  data={filtered}
  keyExtractor={(item) => item.id.toString()}
  renderItem={({ item }) => <Text>{'{item.name}'}</Text>}
/>
// refreshing : booléen qui contrôle le spinner de pull-to-refresh
// (différent de "loading" initial — celui-ci ne cache pas la liste)
const [refreshing, setRefreshing] = useState(false);

// ─── CALLBACK DE RAFRAÎCHISSEMENT ───────────────────────────────────────
// Appelée automatiquement quand l'utilisateur tire la liste vers le bas
const onRefresh = async () => {
  setRefreshing(true);    // ← affiche le spinner de pull-to-refresh
  await fetchData();       // ← re-lance le même fetch qu'au montage
  setRefreshing(false);   // ← cache le spinner (en général géré par finally)
};

<FlatList
  data={items}
  keyExtractor={(item) => item.id.toString()}
  renderItem={({ item }) => <Text>{'{item.name}'}</Text>}
  // refreshControl : prop spéciale qui active le pull-to-refresh
  // Elle attend un composant RefreshControl avec refreshing et onRefresh
  refreshControl={
    <RefreshControl
      // refreshing : true = montrer le spinner, false = le cacher
      refreshing={refreshing}
      // onRefresh : fonction appelée quand l'utilisateur tire vers le bas
      onRefresh={onRefresh}
      // tintColor : couleur du spinner de pull-to-refresh (iOS)
      tintColor="#34d399"
    />
  }
/>

🗂 Architecture de navigation — App Quiz
💡En J2, la commutation AuthStack/AppStack se fait via un booléen isLoggedIn (simulé). En J3, on le remplace par le listener Firebase onAuthStateChanged() — la structure de navigation reste identique.
App.js — commutation AuthStack / AppStack avec booléen mock
// ─── IMPORTS ────────────────────────────────────────────────────────────────
import { useState } from 'react';
import { NavigationContainer } from '@react-navigation/native';
// Nos deux "mondes" de navigation : non-connecté et connecté
import { AuthStack } from './navigation/AuthStack';
import { AppStack  } from './navigation/AppStack';

export default function App() {
  // ─── ÉTAT D'AUTHENTIFICATION (MOCK J2) ─────────────────────────────────
  // false = non connecté → affiche AuthStack (Login/Register)
  // true  = connecté     → affiche AppStack  (Tab avec les écrans de l'app)
  // ⚠ En J3, ce useState sera REMPLACÉ par onAuthStateChanged() de Firebase
  //   mais la logique de rendu conditionnel ci-dessous restera la même
  const [isLoggedIn, setIsLoggedIn] = useState(false);

  return (
    <NavigationContainer>
      {isLoggedIn
        // ─ Si connecté : AppStack avec callback de déconnexion
        // onLogout sera appelé depuis ProfileScreen quand l'utilisateur
        // clique sur "Se déconnecter" → repasse isLoggedIn à false
        ? <AppStack  onLogout={() => setIsLoggedIn(false)} />
        // ─ Si non connecté : AuthStack avec callback de connexion réussie
        // onLogin sera appelé depuis LoginScreen après la validation du form
        : <AuthStack onLogin={() =>  setIsLoggedIn(true)}  />
      }
    </NavigationContainer>
  );
}

10 QCM — Vérification des acquis J2

Navigation · useEffect · Fetch · Architecture

📝Sélectionne une réponse par question puis clique sur "Valider mes réponses".
Question 1 / 10 · Navigation
Quel composant doit impérativement envelopper toute l'application pour que React Navigation fonctionne ?
NavigationContainer est le wrapper obligatoire qui fournit le contexte de navigation à toute l'app. Il ne doit apparaître qu'une seule fois, dans App.js. Sans lui, aucun navigator ne fonctionne — erreur "Couldn't find a navigation object".
Question 2 / 10 · Stack Navigator
Comment passer des paramètres d'un écran A vers un écran B en Stack Navigation ?
Le deuxième argument de navigate() est un objet de paramètres. L'écran destination les reçoit via route.params. C'est le mécanisme standard React Navigation — pas de contexte ni de stockage externe nécessaire.
Question 3 / 10 · Stack vs Tab
Quelle est la différence principale entre Stack Navigator et Tab Navigator pour l'état des écrans en mémoire ?
Dans un Tab, tous les onglets restent montés en mémoire une fois visités — changer d'onglet est instantané. Dans un Stack, les écrans dépilés via goBack() sont détruits. C'est pourquoi les tabs sont plus réactifs que les stacks.
Question 4 / 10 · useEffect
Que signifie le tableau [] (vide) comme deuxième argument de useEffect ?
[] signifie "aucune dépendance" — l'effet s'exécute une seule fois juste après le premier rendu (montage). C'est le pattern standard pour un fetch initial. Sans tableau du tout, l'effet tourne après chaque re-render → risque de boucle infinie.
Question 5 / 10 · useEffect
Pourquoi ne peut-on pas écrire useEffect(async () => {'{ await fetch(...) }'}, []) directement ?
useEffect attend que sa callback retourne soit rien, soit une fonction de cleanup. Une fonction async retourne toujours une Promise — ce qui crée des comportements inattendus. Pattern correct : définir une fonction fetchData async à l'intérieur et l'appeler immédiatement.
Question 6 / 10 · Fetch
Dans quel ordre s'enchaînent les 3 états lors d'un fetch réussi ?
L'ordre : setLoading(true) → afficher spinner → fetch() → setData(json) → setLoading(false) dans le bloc finally → afficher les données. Le finally garantit que le spinner disparaît même si le fetch échoue.
Question 7 / 10 · Expo Router vs React Navigation
Que génère npx create-expo-app MonApp par défaut depuis Expo SDK 49+ ?
Depuis Expo SDK 49, npx create-expo-app génère Expo Router par défaut (dossier app/, fichiers _layout.tsx, TypeScript). Pour un projet React Navigation classique, utiliser --template blank qui donne un App.js simple.
Question 8 / 10 · Fil rouge
En J2, la commutation AuthStack/AppStack est gérée par un booléen mock. En J3, que le remplacera ?
onAuthStateChanged() est le listener Firebase qui écoute les changements d'état d'auth en temps réel. Il retourne l'utilisateur courant (ou null si déconnecté). On remplace isLoggedIn par !!user — la structure de navigation ne change pas.
Question 9 / 10 · Navigation imbriquée
L'utilisateur navigue de HomeScreen → QuizScreen dans l'onglet Home. Il bascule vers l'onglet Profil puis revient sur l'onglet Home. Que se passe-t-il ?
Le Tab Navigator conserve l'état de chaque onglet en mémoire. La Stack de l'onglet Home (avec QuizScreen en haut) est préservée quand l'utilisateur bascule vers Profil. Il retrouve exactement où il en était à son retour.
Question 10 / 10 · Architecture
Quelle est la bonne pratique pour organiser les fichiers de navigation dans AppQuiz ?
Séparer dans /navigation est la convention standard. AppNavigator orchestre la commutation, AuthStack gère les écrans d'auth, AppStack les écrans de l'app connectée. Ça facilite l'évolution : en J3, on modifie uniquement AppNavigator.js pour brancher Firebase.

Glossaire — Termes clés J2

Navigation · useEffect · Fetch · Architecture

🧭 React Navigation

NavigationContainer Navigation
Composant wrapper obligatoire qui fournit le contexte de navigation à toute l'app. Se place une seule fois dans App.js. Sans lui, aucun navigator ne fonctionne.
Équivalent du <BrowserRouter> de React Router web.
Stack Navigator Navigation
Gère une pile d'écrans. navigate() empile un écran, goBack() le dépile. Fournit un header avec bouton retour automatique sur iOS/Android.
Utilisé pour les flux linéaires : liste → détail → sous-détail.
Bottom Tab Navigator Navigation
Barre d'onglets en bas de l'écran. Chaque onglet conserve son état en mémoire une fois monté — passage d'un onglet à l'autre est instantané.
Pattern omniprésent dans les apps mobiles (Instagram, YouTube…).
route.params Navigation
Objet contenant les paramètres passés lors d'un navigate('Ecran', params). Accessible dans le composant destination via la prop route fournie automatiquement.
const {'{'} id, title {'}'} = route.params;
navigation.navigate() Navigation
Navigue vers un écran en l'empilant dans la Stack. Accepte un 2ème argument (objet) pour passer des paramètres. Si l'écran est déjà dans la pile, navigue vers l'instance existante.
Vs replace() qui remplace sans empiler — utile après une connexion réussie.
Navigation imbriquée Navigation
Pattern qui combine Tab + Stack : chaque onglet du Tab a sa propre Stack indépendante. Naviguer "en profondeur" dans un onglet ne change pas l'onglet actif.
Pattern utilisé dans App Quiz : Tab (HomeStack + Profil).

⚡ Hooks & Asynchrone

useEffect Hook React
Hook qui exécute du code après le rendu du composant. Tableau de dépendances : [] = une fois au montage, [id] = à chaque changement de id, pas de tableau = à chaque re-render.
Sert principalement aux appels API, abonnements, timers.
async / await JavaScript
Syntaxe pour gérer les opérations asynchrones lisiblement. await attend la résolution d'une Promise sans bloquer le thread. Doit être dans une fonction déclarée async.
const data = await fetch(url).then(r => r.json())
try / catch / finally JavaScript
Gestion d'erreurs : try = tentative, catch = en cas d'erreur, finally = toujours exécuté. Le finally garantit setLoading(false) même si le fetch échoue.
finally s'exécute même en cas d'erreur → spinner toujours caché.
ActivityIndicator Composant RN
Spinner natif React Native. Afficher pendant loading=true, masquer quand les données arrivent. Natif iOS et Android — apparence différente selon la plateforme.
<ActivityIndicator size="large" color="#34d399" />
RefreshControl Composant RN
Ajoute le geste pull-to-refresh sur FlatList ou ScrollView. Déclenche onRefresh quand l'utilisateur tire vers le bas. Utilise un état refreshing séparé du loading initial.
Combiné avec un état refreshing booléen pour le spinner.
fetch() API Web
API standard JavaScript pour les requêtes HTTP GET/POST/PUT/DELETE. Retourne une Promise. Disponible nativement en React Native — pas besoin d'axios.
const res = await fetch(url); const json = await res.json();

🏗 Architecture App Quiz

AuthStack Architecture
Stack dédiée aux écrans d'authentification (Login, Register). Affichée quand l'utilisateur n'est pas connecté. Remplacée par AppStack à la connexion réussie.
Pattern universel dans toutes les apps avec auth.
AppStack Architecture
Navigator principal (Tab) affiché quand l'utilisateur est connecté. Contient les onglets Home (avec Stack interne) et Profil.
Commutation via isLoggedIn (J2) → onAuthStateChanged (J3).
Mock / Données mockées Architecture
Données ou comportements simulés en attendant l'implémentation réelle. En J2, isLoggedIn est un booléen mock — en J3 Firebase Auth le remplace. La structure ne change pas.
Permet de tester la navigation sans backend.
Expo Router Expo SDK 49+
Système de navigation basé sur les fichiers (comme Next.js). Généré par défaut par npx create-expo-app depuis SDK 49. Utilise le dossier app/ et les fichiers _layout.tsx. Différent de React Navigation.
Pour ce cours : utiliser --template blank pour avoir React Navigation classique.

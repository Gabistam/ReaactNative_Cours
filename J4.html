⌨️ Formulaires & clavier natif
💡 Analogie

Sur mobile, le clavier est une contrainte majeure — il occupe ~40% de l'écran. Sans KeyboardAvoidingView, tes champs de saisie peuvent être cachés sous le clavier. C'est une des erreurs UX les plus fréquentes sur les apps débutantes.

1

KeyboardAvoidingView — éviter que le clavier masque les champs

Enveloppe le formulaire. Sur iOS : behavior="padding". Sur Android : behavior="height" ou géré via le manifest. Le composant remonte automatiquement le contenu quand le clavier apparaît.

2

returnKeyType — personnaliser la touche Entrée

"next" pour passer au champ suivant, "done" pour fermer le clavier, "search" pour les barres de recherche, "send" pour les messages.

3

keyboardType — clavier adapté au contenu

"email-address" pour les emails, "numeric" pour les chiffres, "phone-pad" pour les téléphones. Le bon clavier = moins d'erreurs utilisateur.

4

ref + focus() — navigation entre champs

Associer une ref à chaque TextInput. Dans onSubmitEditing, appeler ref.current.focus() sur le champ suivant. L'utilisateur passe d'un champ à l'autre sans toucher l'écran.

Formulaire complet — Login avec clavier optimisé
import { KeyboardAvoidingView, Platform, TextInput, ScrollView } from 'react-native';
import { useRef } from 'react';

const LoginScreen = () => {
  const passwordRef = useRef(null);

  return (
    <KeyboardAvoidingView
      style={{ flex: 1 }}
      behavior={Platform.OS === 'ios' ? 'padding' : 'height'}
    >
      <ScrollView keyboardShouldPersistTaps="handled">
        <TextInput
          placeholder="Email"
          keyboardType="email-address"
          autoCapitalize="none"
          autoCorrect={false}
          returnKeyType="next"
          onSubmitEditing={() => passwordRef.current.focus()}
        />
        <TextInput
          ref={passwordRef}
          placeholder="Mot de passe"
          secureTextEntry     // ← masque les caractères
          returnKeyType="done"
          onSubmitEditing={handleLogin}
        />
      </ScrollView>
    </KeyboardAvoidingView>
  );
};

💾 AsyncStorage — Persistance locale
💡 Analogie

AsyncStorage, c'est le localStorage du mobile. Un système clé/valeur simple qui persiste entre les redémarrages de l'app. Tout est asynchrone (Promise-based). Idéal pour les préférences légères — pas pour des données volumineuses (utiliser Firestore ou SQLite pour ça).

AsyncStorage — API complète
import AsyncStorage from '@react-native-async-storage/async-storage';

// ÉCRIRE — stocker une valeur (toujours string ou JSON.stringify)
await AsyncStorage.setItem('@theme', 'dark');
await AsyncStorage.setItem('@user_prefs', JSON.stringify({ fontSize: 16, theme: 'dark' }));

// LIRE — récupérer une valeur (peut retourner null si absente)
const theme = await AsyncStorage.getItem('@theme');  // → 'dark' ou null
const prefs = JSON.parse(await AsyncStorage.getItem('@user_prefs') || 'null');

// SUPPRIMER — effacer une clé
await AsyncStorage.removeItem('@theme');

// LISTER — toutes les clés stockées
const keys = await AsyncStorage.getAllKeys();

// VIDER — effacer tout (attention !)
await AsyncStorage.clear();
⚠️AsyncStorage ne stocke que des chaînes. Pour les objets, utiliser JSON.stringify() en écriture et JSON.parse() en lecture. Ne jamais stocker de données sensibles (mots de passe, tokens).
import { useState, useEffect } from 'react';
import AsyncStorage from '@react-native-async-storage/async-storage';

const ThemeToggle = () => {
  const [isDark, setIsDark] = useState(true);

  // Charger le thème sauvegardé au démarrage
  useEffect(() => {
    const loadTheme = async () => {
      const saved = await AsyncStorage.getItem('@theme');
      if (saved !== null) setIsDark(saved === 'dark');
    };
    loadTheme();
  }, []);

  const toggleTheme = async () => {
    const newTheme = !isDark;
    setIsDark(newTheme);
    await AsyncStorage.setItem('@theme', newTheme ? 'dark' : 'light');
  };

  return (
    <TouchableOpacity onPress={toggleTheme}>
      <Text>{isDark ? '🌙 Mode sombre' : '☀️ Mode clair'}</Text>
    </TouchableOpacity>
  );
};

♿ Accessibilité de base
💡L'accessibilité mobile n'est pas une option — c'est une obligation légale dans de nombreux pays et une bonne pratique universelle. React Native expose les APIs d'accessibilité iOS (VoiceOver) et Android (TalkBack) via les mêmes props.
Props d'accessibilité essentielles
// accessibilityLabel — texte lu par le lecteur d'écran
<TouchableOpacity
  accessibilityLabel="Bouton Se connecter"
  accessibilityHint="Lance la connexion avec vos identifiants"
  accessibilityRole="button"
>
  <Text>Se connecter</Text>
</TouchableOpacity>

// accessibilityState — état courant (désactivé, sélectionné...)
<TouchableOpacity
  disabled={!isValid}
  accessibilityState={{ disabled: !isValid }}
>

// Zones tactiles ≥ 44pt (recommandation Apple/Google)
touchTarget: { minWidth: 44, minHeight: 44, justifyContent: 'center', alignItems: 'center' }

// useColorScheme — détecter le thème système
import { useColorScheme } from 'react-native';
const colorScheme = useColorScheme(); // 'dark' | 'light' | null
PropValeursUsage
accessibilityLabelstringTexte lu par VoiceOver/TalkBack
accessibilityHintstringDescription de l'action résultante
accessibilityRolebutton, link, image, header…Rôle sémantique du composant
accessibilityState{'{ disabled, selected, checked… }'}État dynamique du composant
accessibletrue/falseRendre ou non accessible au lecteur

⚙️ Configuration production — app.json
💡 Analogie

app.json, c'est la carte d'identité de ton app. C'est là que tu définis le nom qui s'affiche sur le téléphone, l'icône, le splash screen, et toutes les métadonnées nécessaires au store. Sans une config propre, l'app ne sera pas acceptée par Google Play.

app.json — configuration minimale production
{
  "expo": {
    "name":    "App Quiz",               // ← nom affiché sur l'écran d'accueil
    "slug":    "app-quiz",              // ← identifiant URL (minuscules, tirets)
    "version": "1.0.0",
    "orientation": "portrait",       // portrait | landscape | default
    "icon": "./assets/icon.png",       // ← 1024x1024px, fond opaque
    "splash": {
      "image":           "./assets/splash.png",
      "resizeMode":      "contain",
      "backgroundColor": "#0f172a"
    },
    "android": {
      "adaptiveIcon": {
        "foregroundImage": "./assets/adaptive-icon.png",
        "backgroundColor": "#0f172a"
      },
      "package": "com.tonpseudo.appquiz",  // ← unique — format reverse domain
      "versionCode": 1
    },
    "ios": {
      "bundleIdentifier": "com.tonpseudo.appquiz",
      "buildNumber": "1"
    },
    "extra": {
      "eas": { "projectId": "votre-eas-project-id" }
    }
  }
}
⚠️Le champ package Android doit être unique dans le monde entier. Utiliser le format reverse domain : com.tonpseudo.appquiz. Il ne peut jamais être changé après publication sur le Play Store.
🏗 EAS Build — Générer l'APK Android
💡 Analogie

EAS Build, c'est un serveur de compilation dans le cloud. Tu envoies ton code source, EAS compile l'APK sur ses serveurs (avec les outils natifs Android et iOS installés) et te retourne le fichier binaire. Plus besoin d'Android Studio installé localement.

Installer EAS CLI : npm install -g eas-cli
Se connecter : eas login (compte Expo requis)
Configurer le projet : eas build:configure → génère eas.json
Lancer le build Android preview : eas build -p android --profile preview
Attendre ~5-15 minutes → lien de téléchargement APK dans le terminal
Installer l'APK sur Android : télécharger → activer "sources inconnues" → installer
eas.json — profils de build
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "android": { "buildType": "apk" }  // ← APK direct, pas AAB
    },
    "production": {
      "android": { "buildType": "app-bundle" }  // ← AAB pour Play Store
    }
  }
}
ProfilFormatUsage
developmentAPK dev clientTests avec Expo Dev Client (accès natif)
previewAPK✅ Tests internes, partage avec les profs/équipe
productionAABPublication sur Google Play Store
✅Pour ce cours, utiliser le profil preview — il génère un APK directement installable sans passer par le Play Store. C'est le format attendu pour le rendu J4.

Question 1 / 10 · Formulaires
Sur iOS, quelle valeur de behavior utiliser avec KeyboardAvoidingView ?
Sur iOS, behavior="padding" ajoute du padding en bas pour compenser la hauteur du clavier — c'est le comportement recommandé. Sur Android, behavior="height" fonctionne mieux, ou pas de behavior du tout si le manifest Android est configuré avec windowSoftInputMode.
Question 2 / 10 · Formulaires
Comment passer automatiquement du champ email au champ mot de passe quand l'utilisateur appuie sur "Suivant" ?
La combinaison returnKeyType="next" + useRef + focus() est le pattern standard. returnKeyType="next" change l'icône du bouton Entrée, useRef crée une référence vers le TextInput suivant, et onSubmitEditing déclenche passwordRef.current.focus() quand l'utilisateur appuie sur la touche.
Question 3 / 10 · AsyncStorage
Pourquoi AsyncStorage stocke-t-il uniquement des chaînes de caractères ?
AsyncStorage est un simple store clé/valeur où les valeurs sont des strings — comme localStorage sur le web. Pour stocker un objet : JSON.stringify(monObjet) en écriture, JSON.parse(valeur) en lecture. Toujours gérer le cas null (clé absente).
Question 4 / 10 · AsyncStorage
Dans quel cas AsyncStorage n'est pas adapté au stockage des données ?
AsyncStorage est idéal pour les préférences légères (thème, langue, paramètres). Pour des données volumineuses avec requêtes complexes, utiliser Firestore (données cloud) ou SQLite/expo-sqlite (base locale relationnelle). AsyncStorage ne supporte pas les requêtes, le tri ou les jointures.
Question 5 / 10 · Accessibilité
Quelle est la taille minimale recommandée pour une zone tactile sur mobile (iOS/Android) ?
44pt est la recommandation officielle d'Apple (Human Interface Guidelines) et de Google (Material Design). C'est la taille minimale pour qu'une zone soit confortablement tappable par un doigt adulte. En dessous, le taux d'erreur de tap augmente significativement.
Question 6 / 10 · Accessibilité
À quoi sert la prop accessibilityRole="button" ?
accessibilityRole communique le rôle sémantique au lecteur d'écran. "button" indique que l'élément est cliquable — VoiceOver annonce "Bouton" avant le label. D'autres rôles : "link", "image", "header", "tab". Sans ce rôle, un TouchableOpacity personnalisé peut être lu comme un élément générique non interactif.
Question 7 / 10 · Pressable
Quelle est la différence principale entre Pressable et TouchableOpacity ?
Pressable expose ({ pressed }) dans la prop style. Exemple : style={({ pressed }) => [styles.btn, pressed && styles.btnPressed]}. Cela permet de changer le style en temps réel au press sans JavaScript additionnel. TouchableOpacity ne fait que changer l'opacité — Pressable offre plus de contrôle.
Question 8 / 10 · EAS Build
Quelle différence entre le format APK (profil preview) et AAB (profil production) ?
APK (Android Package) est un fichier binaire complet et autonome — on peut l'envoyer par email et l'installer directement. AAB (Android App Bundle) est un format que Google Play décompose pour générer des APKs optimisés par device (architecture, écran, langue). Play Store exige le AAB depuis 2021.
Question 9 / 10 · EAS Build
Pourquoi le champ package dans app.json ne peut jamais être changé après publication ?
Le package (ex: com.alice.appquiz) est l'identifiant permanent de l'app sur le Play Store. C'est ce que Google utilise pour les mises à jour, les achats in-app, les permissions. Changer le package = publier une nouvelle app from scratch. Les utilisateurs existants restent sur l'ancienne version indéfiniment.
Question 10 / 10 · UX
Pourquoi désactiver le bouton de soumission d'un formulaire pendant une requête Firebase en cours ?
Le double-submit est une des bugs UX les plus fréquents. Sans désactivation, un utilisateur qui tape deux fois sur "S'inscrire" peut créer deux comptes ou envoyer deux requêtes. Pattern : setLoading(true) au début, disabled={loading} sur le bouton, setLoading(false) dans finally.

📱 UX Mobile & Formulaires

KeyboardAvoidingView Composant RN
Composant qui remonte le contenu quand le clavier virtuel apparaît, évitant qu'il masque les champs de saisie. Behavior "padding" sur iOS, "height" sur Android.
Indispensable sur tout écran avec des TextInput.
returnKeyType TextInput
Prop qui personnalise l'icône de la touche Entrée. Valeurs : "next", "done", "search", "send". Améliore le flux de navigation dans un formulaire.
"next" = passer au champ suivant, "done" = fermer le clavier.
secureTextEntry TextInput
Prop booléenne qui masque les caractères saisis (affiche des points). Indispensable pour les champs mot de passe.
Toujours combiner avec un bouton "afficher/masquer".
Pressable Composant RN
Composant interactif plus flexible que TouchableOpacity. Expose ({ pressed }) dans la prop style pour des feedbacks visuels personnalisés en temps réel.
style={'{'}({ pressed }) => [styles.btn, pressed && styles.active]{'}'}
keyboardShouldPersistTaps ScrollView
Prop ScrollView qui contrôle si le clavier se ferme ou non quand l'utilisateur tape en dehors d'un champ. "handled" = recommandé pour les formulaires.
Sans ça, taper sur un bouton ferme d'abord le clavier avant d'exécuter l'action.

💾 Persistance & Accessibilité

AsyncStorage Persistance
Store clé/valeur local, asynchrone, persistant entre les redémarrages. Pour les préférences légères. Stocke uniquement des strings — utiliser JSON.stringify/parse pour les objets.
Équivalent mobile de localStorage.
accessibilityLabel Accessibilité
Texte lu par VoiceOver (iOS) et TalkBack (Android) à la place ou en plus du contenu visuel. Obligatoire sur les éléments sans texte visible (icônes, images).
accessibilityLabel="Fermer la modal"
accessibilityRole Accessibilité
Communique le type sémantique du composant au lecteur d'écran. Valeurs : button, link, image, header, tab, checkbox, radio, menu…
Permet aux utilisateurs de lecteur d'écran de naviguer par type d'élément.
useColorScheme Hook RN
Hook qui retourne le thème système de l'utilisateur : "dark", "light" ou null. Permet d'adapter l'interface au thème préféré du téléphone automatiquement.
const scheme = useColorScheme(); // 'dark' | 'light'

🚀 EAS Build & Déploiement

EAS Build Déploiement
Service cloud Expo qui compile l'app React Native en binaire natif (APK/IPA) sans avoir besoin d'Android Studio ou Xcode en local. Géré via eas-cli.
eas build -p android --profile preview
APK Format Android
Android Package — fichier binaire directement installable sur un appareil Android. Format utilisé pour les tests internes (profil preview EAS). Non optimisé pour le Play Store.
Profil preview EAS : buildType: "apk".
AAB Format Android
Android App Bundle — format requis par Google Play depuis 2021. Google Play le décompose en APKs optimisés par appareil. Plus léger pour l'utilisateur final.
Profil production EAS : buildType: "app-bundle".
app.json Config Expo
Fichier de configuration central d'une app Expo. Définit le nom, l'icône, le splash screen, le package Android, l'identifiant iOS, et toutes les métadonnées du build.
Le champ "package" Android ne peut jamais changer après publication.
eas.json Config EAS
Fichier de configuration EAS Build. Définit les profils (development, preview, production) avec leurs paramètres spécifiques de build.
Généré automatiquement par eas build:configure.

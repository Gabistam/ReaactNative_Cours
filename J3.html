☁️ Qu'est-ce que Firebase ?
💡 Analogie

Firebase, c'est un backend clé en main hébergé par Google. Au lieu de créer toi-même un serveur Node.js, une base de données, un système d'authentification et des APIs, Firebase te fournit tout ça en quelques lignes de configuration. Tu te concentres sur le front — Firebase s'occupe du reste.

ServiceCe qu'il faitQuand l'utiliser
Firebase AuthGère les comptes utilisateurs (email/pass, Google, Apple…)Dès qu'il faut identifier un utilisateur
Cloud FirestoreBase NoSQL en temps réel — documents organisés en collectionsStocker des données structurées accessibles depuis l'app
Realtime DatabaseBase JSON très simple — plus basique que FirestoreCas simples, chat, données très plates
StorageHébergement de fichiers (images, vidéos, PDFs)Upload d'avatars, photos…
📦 Installation & Configuration du projet Firebase
Aller sur console.firebase.google.com → Créer un projet → Désactiver Google Analytics (pas nécessaire)
Dans le projet Firebase → icône </> → Ajouter une application Web (pas iOS/Android) → Enregistrer
Copier les clés de configuration affichées → elles serviront dans firebase.config.js
Dans la console Firebase → Authentication → Commencer → Activer le fournisseur Email/Mot de passe
Installer le SDK Firebase dans le projet : npm install firebase
Créer le fichier services/firebase.js avec la configuration (voir code ci-dessous)

// ─── IMPORTS SDK FIREBASE ──────────────────────────────────────────────────
import { initializeApp }                  from 'firebase/app';
import { getAuth, initializeAuth,
         getReactNativePersistence }       from 'firebase/auth';
import { getFirestore }                  from 'firebase/firestore';
import AsyncStorage                       from '@react-native-async-storage/async-storage';

const firebaseConfig = {
  apiKey:            "AIzaSy...",
  authDomain:        "ton-projet.firebaseapp.com",
  projectId:         "ton-projet",
  storageBucket:     "ton-projet.appspot.com",
  messagingSenderId: "123456789",
  appId:             "1:123456789:web:abc...",
};

const app = initializeApp(firebaseConfig);

const auth = initializeAuth(app, {
  persistence: getReactNativePersistence(AsyncStorage),

🔐 Authentification email / mot de passe
💡 Analogie

Firebase Auth, c'est comme un videur de boîte de nuit. Il vérifie que l'email existe, que le mot de passe est correct, et délivre un "bracelet" (le token JWT). Ce bracelet est envoyé automatiquement à chaque requête Firestore — tu n'as pas à gérer les tokens toi-même.

1

createUserWithEmailAndPassword(auth, email, password) — inscription

Crée un nouveau compte dans Firebase Auth. Retourne une Promise avec un UserCredential.

2

signInWithEmailAndPassword(auth, email, password) — connexion

Tente de connecter l'utilisateur. Si incorrects → lance une erreur (auth/wrong-password ou auth/user-not-found).

3

signOut(auth) — déconnexion

Déconnecte l'utilisateur courant. Le listener onAuthStateChanged sera notifié avec null.

4

onAuthStateChanged(auth, callback) — écouter l'état d'auth

Appelé automatiquement quand l'utilisateur se connecte, se déconnecte, ou au démarrage. C'est lui qui remplace le booléen mock de J2.

Exemples — cliquer pour développer :

import { useState } from 'react';
import { View, Text, TextInput, TouchableOpacity, Alert, ActivityIndicator } from 'react-native';
import { auth } from '../../services/firebase';
import { createUserWithEmailAndPassword } from 'firebase/auth';

export default function RegisterScreen() {
  const [email, setEmail]       = useState('');
  const [password, setPassword] = useState('');
  const [loading, setLoading]   = useState(false);

  const handleRegister = async () => {
    if (!email || !password) { Alert.alert('Erreur', 'Remplis tous les champs.'); return; }
    try {
      setLoading(true);
      const userCredential = await createUserWithEmailAndPassword(auth, email, password);
      console.log('Compte créé :', userCredential.user.uid);
    } catch (err) {
      Alert.alert('Erreur', err.message);
    } finally { setLoading(false); }
  };

  return (
    <View>
      <TextInput value={email} onChangeText={setEmail} placeholder="Email" keyboardType="email-address" autoCapitalize="none" />
      <TextInput value={password} onChangeText={setPassword} placeholder="Mot de passe" secureTextEntry />
      <TouchableOpacity onPress={handleRegister} disabled={loading}>
        {loading ? <ActivityIndicator /> : <Text>S'inscrire</Text>}
      </TouchableOpacity>
    </View>
  );
}
export function getErrorMessage(code) {
  const messages = {
    'auth/email-already-in-use':    'Cet email est déjà utilisé.',
    'auth/invalid-email':           'Format d\'email invalide.',
    'auth/weak-password':           'Mot de passe trop court (6 caractères min).',
    'auth/user-not-found':          'Aucun compte avec cet email.',
    'auth/wrong-password':          'Mot de passe incorrect.',
    'auth/too-many-requests':       'Trop de tentatives. Réessaie plus tard.',
    'auth/network-request-failed':  'Problème de connexion internet.',
  };
  return messages[code] ?? 'Une erreur est survenue. Réessaie.';
}
// App.js — APRÈS (J3, Firebase)
import { useState, useEffect } from 'react';
import { auth } from './services/firebase';
import { onAuthStateChanged } from 'firebase/auth';

export default function App() {
  const [user, setUser] = useState(undefined);

  useEffect(() => {
    const unsubscribe = onAuthStateChanged(auth, (firebaseUser) => {
      setUser(firebaseUser);
    });
    return unsubscribe;
  }, []);

  if (user === undefined) return <SplashScreen />;

  return (
    <NavigationContainer>
      {!!user ? <AppStack /> : <AuthStack />}
    </NavigationContainer>
  );
}

🗂 Interaction entre fichiers — Comment tout s'articule ?
🏗 Architecture

Firebase ne s'insère pas dans un seul fichier — il est réparti sur plusieurs couches. Comprendre qui fait quoi évite de mélanger la logique Firebase avec les composants visuels, et permet de réutiliser facilement les fonctions d'auth dans toute l'app.

1

services/firebase.js — La source unique de vérité Firebase

Initialise Firebase une seule fois et exporte les instances auth et db. Tous les autres fichiers importent depuis ici — jamais ils n'appellent initializeApp() eux-mêmes. C'est le seul fichier qui connaît les clés de configuration.

2

services/authService.js — Les fonctions Auth centralisées

Importe auth depuis firebase.js et expose des fonctions simples : signUp, signIn, logOut. Les écrans n'importent jamais le SDK Firebase directement — ils passent par ce service. Avantage : si Firebase change d'API, on ne modifie qu'un seul fichier.

3

App.js — Le chef d'orchestre (pièce clé 🔑)

Importe auth et écoute onAuthStateChanged. Quand l'état change, il met à jour user en state. C'est ce state qui décide quel Navigator afficher. Ni LoginScreen, ni HomeScreen n'ont besoin de naviguer manuellement — App.js réagit automatiquement à chaque changement d'état.

4

navigation/AppNavigator.js — L'aiguilleur Auth ↔ App

Reçoit user en prop depuis App.js. Si user est un objet → affiche AppStack. Si null → affiche AuthStack. Ce composant ne fait rien d'autre — logique minimale, impact maximal.

5

screens/LoginScreen.js — Utilise le service, ignore Firebase

Importe uniquement signIn() depuis authService.js. Après l'appel, ne navigue jamais manuellement — Firebase notifie App.js via onAuthStateChanged, App.js met à jour user, AppNavigator bascule vers AppStack. Tout est automatique et réactif.

🔄 Le flux complet pas-à-pas

①

Utilisateur saisit email + mot de passe → appuie sur "Se connecter"

LoginScreen.handleLogin() est appelé.

②

authService.signIn(email, password) est appelé

Qui appelle en interne signInWithEmailAndPassword(auth, email, password) du SDK Firebase.

③

Le SDK Firebase contacte les serveurs Google

Si les identifiants sont corrects → Firebase génère un token JWT et le sauvegarde dans AsyncStorage.

④

onAuthStateChanged se déclenche automatiquement dans App.js

Firebase notifie tous ses listeners actifs. setUser(firebaseUser) est appelé avec l'objet User.

⑤

React re-rend App.js → !!user vaut true

AppNavigator reçoit user ≠ null → affiche AppStack. L'écran Login disparaît automatiquement — sans navigation.navigate().

💡La beauté du pattern réactif : LoginScreen n'a jamais besoin de naviguer. Il fait son travail (appeler Firebase), et c'est l'état user dans App.js qui pilote toute la navigation. C'est le principe "state-driven navigation" — l'UI suit l'état, pas l'inverse.

Code des fichiers clés — cliquer pour développer :

// ─── services/authService.js ─────────────────────────────────────────────────
// Centralise TOUTES les opérations Firebase Auth
// Les écrans importent ces fonctions — jamais le SDK Firebase directement
// Avantage : si Firebase change d'API, on modifie un seul fichier

import {
  createUserWithEmailAndPassword,
  signInWithEmailAndPassword,
  signOut
} from 'firebase/auth';
// On importe l'instance auth depuis firebase.js — jamais on ne la recrée
import { auth } from './firebase';

// ── Inscription ──────────────────────────────────────────────────────────────
// Retourne une Promise avec UserCredential si succès
// Lance une erreur avec err.code si échec (email déjà pris, mdp trop court…)
export const signUp = (email, password) => {
  return createUserWithEmailAndPassword(auth, email, password);
};

// ── Connexion ────────────────────────────────────────────────────────────────
export const signIn = (email, password) => {
  return signInWithEmailAndPassword(auth, email, password);
};

// ── Déconnexion ──────────────────────────────────────────────────────────────
// Après l'appel, onAuthStateChanged reçoit null → App.js bascule vers AuthStack
export const logOut = () => {
  return signOut(auth);
};
// ─── App.js ──────────────────────────────────────────────────────────────────
// Point d'entrée de l'app. Son seul rôle Firebase : écouter l'état d'auth
// et passer l'info à AppNavigator. Toute la navigation découle de ce state.

import { useState, useEffect }    from 'react';
import { NavigationContainer }    from '@react-navigation/native';
import { ActivityIndicator, View } from 'react-native';
import { onAuthStateChanged }      from 'firebase/auth';
import { auth }                    from './services/firebase';
import AppNavigator               from './navigation/AppNavigator';

export default function App() {
  // undefined → Firebase n'a pas encore répondu (état initial)
  // null      → déconnecté | objet User → connecté
  const [user, setUser] = useState(undefined);

  useEffect(() => {
    // onAuthStateChanged s'exécute :
    // 1) Immédiatement au montage (vérifie la session AsyncStorage)
    // 2) À chaque connexion / déconnexion
    const unsubscribe = onAuthStateChanged(auth, (firebaseUser) => {
      setUser(firebaseUser);
    });
    return unsubscribe; // cleanup → évite fuite mémoire
  }, []);

  // Spinner pendant que Firebase vérifie la session
  // Sans ça : flash de AuthStack avant confirmation de session
  if (user === undefined) {
    return (
      <View style={'{'}flex:1,justifyContent:'center',alignItems:'center'{'}'}>
        <ActivityIndicator size="large" />
      </View>
    );
  }

  return (
    <NavigationContainer>
      // !!user = true si objet (connecté), false si null (déconnecté)
      <AppNavigator user={user} />
    </NavigationContainer>
  );
}
// ─── screens/auth/LoginScreen.js ─────────────────────────────────────────────
// Ce screen NE CONNAÎT PAS Firebase directement — il utilise authService
// Il NE NAVIGUE PAS manuellement — App.js gère tout via onAuthStateChanged

import { useState }                   from 'react';
import { View, Text, TextInput,
         TouchableOpacity, Alert,
         ActivityIndicator }         from 'react-native';
import { signIn }         from '../../services/authService';
import { getErrorMessage } from '../../utils/firebaseErrors';

export default function LoginScreen() {
  const [email,    setEmail]    = useState('');
  const [password, setPassword] = useState('');
  const [loading,  setLoading]  = useState(false);

  const handleLogin = async () => {
    if (!email || !password) {
      Alert.alert('Erreur', 'Remplis tous les champs.'); return;
    }
    setLoading(true);
    try {
      await signIn(email, password);
      // ✅ Pas de navigation ici !
      // Firebase → onAuthStateChanged → setUser() → AppNavigator → AppStack
      // La bascule est 100% automatique et réactive
    } catch (err) {
      Alert.alert('Connexion échouée', getErrorMessage(err.code));
    } finally {
      setLoading(false);
    }
  };

  return (
    <View>
      <TextInput value={email}    onChangeText={setEmail}
        placeholder="Email" keyboardType="email-address" autoCapitalize="none" />
      <TextInput value={password} onChangeText={setPassword}
        placeholder="Mot de passe" secureTextEntry />
      <TouchableOpacity onPress={handleLogin} disabled={loading}>
        {loading ? <ActivityIndicator /> : <Text>Se connecter</Text>}
      </TouchableOpacity>
    </View>
  );
}

Firebase Auth · Firestore · Règles de sécurité · Architecture

📝Sélectionne une réponse par question puis clique sur "Valider mes réponses".
Question 1 / 10 · Firebase Auth
Quelle fonction Firebase permet de créer un nouveau compte avec email et mot de passe ?
createUserWithEmailAndPassword() est la fonction officielle Firebase. Elle retourne une Promise avec UserCredential contenant user.uid, user.email etc.
Question 2 / 10 · onAuthStateChanged
Pourquoi faut-il retourner la fonction unsubscribe dans le cleanup du useEffect contenant onAuthStateChanged ?
Sans unsubscribe, le listener Firebase continue d'écouter même si le composant a été détruit → memory leaks et mises à jour d'état sur un composant démonté.
Question 3 / 10 · Persistance de session
Pourquoi utilise-t-on initializeAuth avec getReactNativePersistence(AsyncStorage) ?
Sans persistance AsyncStorage, le token de session est perdu à chaque fermeture. L'utilisateur doit se reconnecter à chaque lancement.
Question 4 / 10 · Firestore
Dans Firestore, quelle est la différence entre setDoc et addDoc ?
setDoc avec un ID choisi (ex: uid). addDoc = ID aléatoire généré par Firestore, utile pour les sous-collections d'historique.
Question 5 / 10 · Firestore
Comment charger uniquement les questions d'un thème précis ?
query() + where() filtrent les documents. where('themeId', '==', themeId) = équivalent d'un WHERE SQL.
Question 6 / 10 · getDocs vs onSnapshot
Quelle est la différence fondamentale entre getDocs et onSnapshot ?
getDocs = one-shot. onSnapshot = listener permanent appelé à chaque modification. Idéal pour l'historique et les données dynamiques.
Question 7 / 10 · Règles de sécurité
Que signifie request.auth != null dans les règles Firestore ?
request.auth est null si non connecté. != null = l'utilisateur est connecté via Firebase Auth. Condition minimale pour protéger les données.
Question 8 / 10 · Règles de sécurité
Comment s'assurer qu'un utilisateur n'accède qu'à ses propres données dans users/{userId} ?
request.auth.uid = uid de celui qui fait la requête. En le comparant avec userId du chemin, chaque utilisateur n'accède qu'à son propre document.
Question 9 / 10 · setDoc vs updateDoc
Tu veux modifier uniquement le pseudo sans toucher à l'email et la date de création. Quelle fonction ?
updateDoc() modifie seulement les champs spécifiés. setDoc() sans option remplace le document entièrement — les autres champs seraient perdus.
Question 10 / 10 · Architecture
Après signOut(auth), qu'est-ce qui gère automatiquement la redirection vers AuthStack ?
signOut() → onAuthStateChanged reçoit null → setUser(null) → !!user = false → AuthStack. Pas de navigation.replace() nécessaire. L'état pilote l'affichage.

Glossaire — Termes clés J3

Firebase Auth · Firestore · Sécurité · Architecture

🔐 Firebase Auth

Firebase Auth Firebase
Service Firebase qui gère l'authentification. Stocke les comptes, génère les tokens JWT et gère les sessions.
Penser "videur de boîte" : vérifie l'identité, délivre un bracelet (token).
createUserWithEmailAndPassword Auth
Crée un nouveau compte Firebase. Retourne une Promise avec UserCredential. Lance une erreur si l'email est déjà utilisé ou mot de passe trop court.
Après l'appel : user.uid disponible pour créer le profil Firestore.
onAuthStateChanged Auth
Listener temps réel qui notifie à chaque changement d'état d'auth. Reçoit un objet User (connecté) ou null (déconnecté).
Toujours unsubscribe dans le cleanup de useEffect.
UserCredential Auth
Objet retourné par createUser() et signIn(). Contient .user avec uid, email, displayName…
const { user } = await signInWithEmailAndPassword(...)
auth.currentUser Auth
Propriété synchrone qui retourne l'utilisateur connecté, ou null.
auth.currentUser?.uid — opérateur ?. pour éviter les crash si null.
Persistance de session Auth
Mécanisme qui sauvegarde le token entre les relances de l'app. En React Native, on utilise AsyncStorage via getReactNativePersistence().
Sans persistance : reconnexion à chaque ouverture.

🗄 Cloud Firestore

Collection Firestore
Conteneur de documents — l'équivalent d'une table SQL. Ex : themes, questions, users.
collection(db, 'themes') — crée une référence.
Document Firestore
Unité de stockage — l'équivalent d'une ligne SQL. Contient des champs. Max 1 Mo.
doc(db, 'users', uid) — référence vers un document précis.
getDocs Firestore
Requête one-shot qui récupère tous les documents. À utiliser pour les données qui changent rarement.
const snap = await getDocs(collection(db, 'themes'))
onSnapshot Firestore
Listener temps réel. Callback appelée immédiatement puis à chaque modification. Retourne unsubscribe.
Idéal pour l'historique, classement, données dynamiques.
serverTimestamp Firestore
Génère un horodatage côté serveur Firebase. Garantit la cohérence entre tous les appareils.
createdAt: serverTimestamp() — toujours préférer au new Date()
QuerySnapshot Firestore
Résultat de getDocs(). Contient .docs de DocumentSnapshot. Chaque doc a .id et .data().
snap.docs.map(doc => ({ id: doc.id, ...doc.data() }))

🛡 Sécurité & Architecture

Règles de sécurité Firestore Sécurité
Langage déclaratif qui définit qui peut lire/écrire dans Firestore. Côté serveur — ne peuvent pas être contournées par le client.
allow read: if request.auth != null;
request.auth.uid Sécurité
Dans les règles, l'uid de l'utilisateur qui fait la requête. À comparer avec l'ID du document pour isoler les données par utilisateur.
request.auth.uid == userId
updateDoc vs setDoc Firestore
updateDoc modifie seulement les champs spécifiés. setDoc remplace le document entièrement. setDoc avec {merge:true} fusionne.
Pour modifier un pseudo : updateDoc, pas setDoc.

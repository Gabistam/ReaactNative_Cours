Contexte du sujet

Mise en situation client · Objectif de la mini-app

🎯 Mise en situation

L'organisme de formation WebCraft Academy souhaite disposer d'une application mobile permettant à ses apprenants et son équipe pédagogique de consulter, en mobilité, les projets en cours de l'école ainsi que le catalogue de formations proposé. L'accès à l'application doit être réservé aux utilisateurs authentifiés.

💼 Tu es le développeur mobile en charge de produire un prototype fonctionnel de cette application, à présenter à l'équipe pédagogique en fin de session.
📋 Sujet — ce que l'application doit faire
  • Un écran d'authentification (connexion / inscription) via Firebase Auth, avant tout accès au contenu de l'app
  • Une fois connecté, l'utilisateur accède à une navigation par onglets (Bottom Tabs) avec au minimum 3 écrans
  • L'écran Liste des projets doit exploiter l'API 1 et permettre d'accéder à un écran de détail au clic sur un projet (navigation imbriquée Stack dans le Tab)
  • Un second écran doit exploiter l'API 2 (formations) — structure libre, à toi de proposer une UX pertinente
  • Les deux APIs doivent être appelées avec une gestion d'erreur try/catch — c'est une obligation, pas une option
⚠️ La structure exacte du 3ᵉ écran (et au-delà) ainsi que le niveau de finition de l'Auth Firebase sont laissés à ton appréciation. Une appli simple mais entièrement fonctionnelle vaut mieux qu'une appli ambitieuse mais cassée.
🧰 Stack imposée

Tu dois utiliser exclusivement les briques vues pendant le module — aucune librairie tierce non enseignée.

  • React Native + Expo (SDK vu en cours)
  • React Navigation v7 — native-stack + bottom-tabs
  • Firebase JS SDK — Auth (email/mot de passe)
  • fetch() natif pour la consommation des deux APIs (pas d'axios)
  • Architecture à plat : pas de dossier services/ ni hooks/ — la logique reste dans les écrans

Architecture de navigation attendue

AppNavigator · AuthStack · AppStack · Bottom Tabs

🧭 Schéma de navigation

Rappel du principe vu en cours : AppNavigator est l'orchestrateur qui bascule entre AuthStack (non connecté) et AppStack (connecté) selon l'état d'authentification Firebase.

Arborescence cible
AppNavigator (orchestrateur — bascule selon état Auth)
│
├── AuthStack (utilisateur non connecté)
│     ├── LoginScreen
│     └── SignupScreen
│
└── AppStack → BottomTabNavigator (utilisateur connecté)
      ├── Tab "Accueil"      → HomeScreen
      ├── Tab "Projets"      → ProjectsStackNavigator
      │                            ├── ProjectsListScreen   (API 1)
      │                            └── ProjectDetailScreen  (détail au clic)
      └── Tab "Formations"   → CoursesScreen (API 2 — structure libre)
✅ Le point clé évalué ici : la Stack imbriquée dans le Tab "Projets" doit fonctionner — liste → détail → retour, sans casser la barre de tabs.
🔐 Logique d'authentification minimale
  • Un état global (ex: useState + onAuthStateChanged) détermine si l'utilisateur est connecté
  • Tant qu'il n'est pas connecté → impossible d'accéder aux Tabs (route protégée de fait, via l'AppNavigator)
  • Niveau d'exigence : Auth Firebase basique suffit (inscription + connexion). Déconnexion et gestion d'erreurs de saisie sont un plus, mais ne sont pas obligatoires.

Ressources fournies — 2 APIs REST

Données à fetch obligatoirement avec gestion d'erreur

⚠️ Obligation : charger et exploiter les DEUX APIs ci-dessous avec gestion d'erreur (try/catch). C'est un critère noté indépendamment de la navigation.
API 1
Projets
Endpoint
GET https://webcraft-api.vercel.app/data/projects.json
Retourne : projects[], technologies[], categories[]
Utilisée pour l'écran Liste + Détail des projets
API 2
Formations
Endpoint
GET https://webcraft-api.vercel.app/data/courses.json
Retourne : courses[], categories[], levels[], technologies[]
Utilisée pour le 2ᵉ écran de contenu (structure libre)
🛡️ Rappel — pattern de fetch attendu

Le try/catch est obligatoire. La gestion visuelle d'un état loading/erreur à l'écran est un bonus, pas une obligation.

Squelette minimal exigé
useEffect(() => {
  const fetchProjects = async () => {
    try {
      const response = await fetch('https://webcraft-api.vercel.app/data/projects.json');
      const data = await response.json();
      setProjects(data.projects);
    } catch (error) {
      // Obligatoire : capturer l'erreur réseau / parsing
      console.log('Erreur API Projets :', error);
    }
  };
  fetchProjects();
}, []);

Consignes détaillées

Obligatoire vs bonus — par écran

Écran 1
Authentification (AuthStack)
Obligatoire
Accès à l'application conditionné par la connexion Firebase.
  • Formulaire de connexion (email + mot de passe) relié à Firebase Auth
  • Formulaire d'inscription (peut être un second écran ou un toggle sur le même écran)
  • Navigation automatique vers l'AppStack une fois connecté
⭐ Bonus possible : bouton de déconnexion accessible, messages d'erreur de saisie (email invalide, mot de passe trop court…).
Écran 2
Accueil (Tab)
Obligatoire
Écran d'accueil de l'application — contenu libre.
  • Doit exister comme tab distinct et être fonctionnel (pas de crash à l'ouverture)
  • Contenu libre : message de bienvenue, résumé, raccourcis vers les autres tabs… à toi de juger
Écran 3
Liste des projets + Détail (Tab → Stack imbriqué)
Obligatoire
Le cœur technique de l'évaluation — navigation Stack imbriquée dans un Tab.
  • Fetch de l'API 1 au chargement de l'écran (useEffect + try/catch)
  • Affichage de la liste des projets via FlatList (pas de .map() dans un ScrollView)
  • Au clic sur un projet → navigation vers un écran ProjectDetailScreen affichant les informations complètes (passage de paramètres via route.params ou re-fetch par id)
  • Le retour à la liste depuis le détail doit fonctionner correctement (header natif ou bouton retour)
⭐ Bonus possible : état de chargement (spinner/texte "Chargement…") et message d'erreur visible à l'écran en cas d'échec du fetch.
Écran 4
Formations (Tab)
Obligatoire
Exploitation de la seconde API — structure laissée libre.
  • Fetch de l'API 2 au chargement de l'écran (try/catch obligatoire)
  • Affichage des formations sous une forme de ton choix (liste simple, cards, regroupement par catégorie/niveau…)
  • Pas d'obligation de navigation vers un détail ici — la simple exploitation des données suffit
⭐ Bonus possible : filtrage ou regroupement par categories/levels/technologies, recherche de l'écran de détail pour ce module également.
Transversal
Qualité de code & UX mobile
Obligatoire
  • Aucun crash pendant la démonstration (un crash ponctuel géré gracieusement vaut mieux qu'un écran blanc)
  • Styles via StyleSheet.create() — pas de styles inline en dur partout
  • Zones tactiles confortables, lisibilité correcte sur mobile
  • Code commenté de manière pertinente (pas nécessaire ligne à ligne, mais les choix clés expliqués)
